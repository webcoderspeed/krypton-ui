## API Reference

Detailed API reference for the Checkbox component and its props.

### Checkbox

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `checked` | `boolean \| "indeterminate"` | - | The controlled checked state of the checkbox. |
| `defaultChecked` | `boolean` | - | The default checked state when initially rendered. |
| `onCheckedChange` | `(checked: boolean \| "indeterminate") => void` | - | Event handler called when the checked state changes. |
| `disabled` | `boolean` | `false` | When true, prevents the user from interacting with the checkbox. |
| `required` | `boolean` | `false` | When true, indicates that the user must check the checkbox before the owning form can be submitted. |
| `name` | `string` | - | The name of the checkbox. Submitted with its owning form as part of a name/value pair. |
| `value` | `string` | `"on"` | The value given as data when submitted with a name. |
| `id` | `string` | - | The id of the checkbox. |
| `className` | `string` | - | Additional CSS classes to apply to the checkbox. |
| `...props` | `React.ComponentProps<"button">` | - | All other props are passed to the underlying button element. |

The checkbox component follows WAI-ARIA guidelines:

- Uses proper ARIA attributes for screen readers
- Supports keyboard navigation (Space to toggle)
- Provides focus indicators
- Supports disabled state
- Works with form labels

### Keyboard Interactions

| Key | Description |
| --- | --- |
| `Space` | Toggles the checkbox between checked and unchecked |
| `Tab` | Moves focus to the next focusable element |
| `Shift + Tab` | Moves focus to the previous focusable element |

## Best Practices

### Do's

- Always provide a clear label for the checkbox
- Use checkboxes for binary choices (yes/no, on/off)
- Group related checkboxes together
- Use indeterminate state for partial selections
- Provide feedback when checkbox state changes

### Don'ts

- Don't use checkboxes for mutually exclusive options (use radio buttons instead)
- Don't make checkbox labels too long
- Don't use checkboxes without labels
- Don't disable checkboxes without explanation

## Advanced Usage

### Indeterminate State

The checkbox supports an indeterminate state, useful for "select all" scenarios:

```tsx
const [checkedItems, setCheckedItems] = useState([false, false, false])

const allChecked = checkedItems.every(Boolean)
const isIndeterminate = checkedItems.some(Boolean) && !allChecked

<Checkbox
  checked={allChecked}
  onCheckedChange={(checked) => {
    setCheckedItems(checkedItems.map(() => checked === true))
  }}
  // Note: indeterminate state is handled automatically by Radix
/>
```

## Troubleshooting

### Common Issues

1. **Checkbox not responding to clicks**
   - Ensure the checkbox is not disabled
   - Check if there are any overlapping elements
   - Verify event handlers are properly attached

2. **Styling not applying**
   - Check if Tailwind CSS is properly configured
   - Ensure CSS classes are not being overridden
   - Verify the component is importing styles correctly

3. **Form submission issues**
   - Make sure the checkbox has a `name` attribute for form submission
   - Check if the form library is properly handling checkbox values
   - Verify the `value` prop is set correctly

4. **Accessibility concerns**
   - Always associate checkboxes with labels using `htmlFor` and `id`
   - Provide clear, descriptive labels
   - Test with screen readers and keyboard navigation

### Performance Considerations

- Use `defaultChecked` for uncontrolled checkboxes when possible
- Avoid unnecessary re-renders by memoizing event handlers
- Consider using `useCallback` for complex `onCheckedChange` handlers

```tsx
const handleChange = useCallback((checked: boolean | "indeterminate") => {
  // Handle change logic
}, [dependencies])

<Checkbox onCheckedChange={handleChange} />
```
